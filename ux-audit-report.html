Top priorities

Desktop app + product website

A polished utility with three trust leaks.

NetReceipt is focused, visually coherent, and unusually clear for a network tool. The highest-value improvements are not aesthetic: they are about ensuring the interface never presents stale evidence as current, never overstates what one reference host proves, and tells one consistent privacy story.

What already works

Keep the restraint.

The interface succeeds because it does not feel like a generic analytics dashboard. It leads with a single answer, keeps the data hierarchy tight, and makes advanced detail available without crowding the default view.

Status, three checks, recent trend, then actions: the order matches the user’s likely questions and works in one compact glance.

Prioritized findings

Fix confidence gaps first.

Severity reflects likely user harm and reach, not implementation difficulty. “High” findings can change whether users trust the evidence or the privacy claim; “Medium” findings create interpretation or completion friction; “Low” findings are accessibility and navigation polish.

NR-UX-01High

Paused monitoring can still look healthy and freshly checked

Desktop app

Pause changes the small header label and footer action, but the dominant green hero and its health claim still come from the last sample. “Checked just now” is calculated only during a render, so it can remain frozen after monitoring stops.

Evidence

src/App.tsx:108–130, 153–155, 290–292. The hero ignores monitor.running; no timer refreshes relative age.

Recommendation

Give “paused” and “stale” higher display precedence than the last sample. Use a neutral hero: “Monitoring paused · Last checked 9:42 PM,” with a clear Resume action.

NR-UX-02High

The website loads Google Analytics before offering consent

Website + privacy

The public page says optional telemetry waits for consent, while its head immediately loads and configures the Google tag. The desktop app does wait for an explicit choice, so the two surfaces tell incompatible privacy stories.

Evidence

docs/index.html:20–27, 37 loads GA unconditionally. In contrast, src/analytics.ts:26–51 gates the app script behind consent.

Recommendation

Default analytics storage to denied and do not load gtag until the visitor opts in. Name Google Analytics in the disclosure and offer equally clear “Allow” and “No thanks” actions.

NR-UX-03High

One reference host is presented as the health of “your internet”

Product model + copy

TCP, DNS, and HTTPS are distinct layers, but all checks target the same configurable host. A host-specific problem can be described as a whole-internet interruption, and the website calls the checks “independent.”

Evidence

src-tauri/src/monitor.rs:102–117 classifies the combined result; the dashboard says “Your internet is healthy/interrupted,” and docs/index.html:37 says “Three independent checks.”

Recommendation

Near term, say “Reference checks passed” and “All checks to one.one.one.one succeeded.” Call them “three connection-layer checks.” Longer term, use independent endpoints before making whole-internet claims.

NR-UX-04Medium

“Issues” and “Incidents” use different counting models

Desktop app

The main chart’s “Issues” value counts every non-healthy sample. History separately reports confirmed incidents after applying failure tolerance. Users can reasonably assume these are two views of the same concept.

Evidence

src/components.tsx:35–55, 79–82 counts samples. src/history-dialog.tsx:58–73 labels a different aggregate as confirmed incidents.

Recommendation

Either calculate confirmed 30-minute incidents for the main card, or rename it “Affected checks” and expose the sample definition in the accessible chart summary.

NR-UX-05Medium

Settings expose implementation terms without decision support

Desktop app

Interval, timeout, slow threshold, retention, and consecutive-issue tolerance are valid controls, but the screen mostly states units. Users must understand monitoring mechanics to predict the effect of a change.

Evidence

The 420 × 640 test fit all fields, but inputs were 30 px tall and explanation was limited to “After sustained changes.” Validation appears after Save.

Recommendation

Add outcome-oriented helper text and safe presets: “Balanced,” “Fast detection,” and “Low activity.” Show the estimated alert delay, e.g. “3 issues × 30 sec ≈ 90 sec.” Add “Restore defaults.”

NR-UX-06Medium

Important status changes are not announced programmatically

Accessibility

Error/success notices correctly use live semantics, but the “Live / Paused” state and connection headline are plain content. A screen-reader user may not hear a monitoring or connection state transition.

Evidence

src/App.tsx:108–130 has no role="status" or live region. Visual state is clear and focus outlines are implemented.

Recommendation

Announce only meaningful transitions with a polite live region, not every sample. Include monitoring state and last-check age in the same concise status sentence.

NR-UX-07Medium

The download decision lacks release confidence cues

Website

Platform choices are well separated, but “latest build” links show no version, publish date, file size, signing/notarization status, or checksum. Those details matter for a background utility with local network access.

Evidence

docs/index.html:38 provides direct asset links and OS requirements only. Release notes are one additional link below all cards.

Recommendation

Show “v0.1.1 · published Jul 2026,” package size, and signing status next to the chooser. Keep checksums and release notes one click away.

NR-UX-08Low

Mobile removes all navigation beyond the wordmark

Website

At 560 px and below, the entire nav link group is hidden. The primary hero CTA still reaches downloads, but return visitors lose direct access to Downloads, Features, GitHub, and Ko-fi.

Evidence

docs/styles.css uses nav>div{display:none} at the mobile breakpoint. At 390 px the page had no horizontal overflow.

Recommendation

Keep a compact “Downloads” link visible beside the wordmark; move lower-priority links to the footer. Add a skip link for keyboard users.

NR-UX-09Low

Two small website text styles miss normal-text contrast

Accessibility

Most app and website text has generous contrast. The 9 px proof strip and download-card secondary link text are the exceptions in sampled color pairs.

Evidence

#647173 on #0d1213 measured 3.73:1; #748182 on #121819 measured 4.45:1. Both are below 4.5:1 for normal text.

Recommendation

Raise those colors to at least #7f8b8c/#81908d and increase the proof strip from 9 px to 10–11 px. Recheck hover and visited states.

Recommended roadmap

Trust, then meaning, then polish.

This sequence keeps the work outcome-focused. The first phase protects the core promise; the second aligns terminology and decision support; the third closes accessibility and acquisition gaps.

Now · 1–2 days

Protect the promise

  1. Add paused and stale states.Neutralize the hero and keep last-check age live.
  2. Gate website analytics.Make the website match the desktop app’s explicit consent model.
  3. Narrow the health claim.Describe reference-host checks accurately on both surfaces.
Next · 3–5 days

Clarify the evidence

  1. Align Issues and Incidents.Use one definition or name the distinct counts plainly.
  2. Guide configuration.Add presets, estimated alert delay, and restore defaults.
  3. Improve download confidence.Add version, date, size, signing, and release-note context.
Then · 1–2 days

Close accessibility gaps

  1. Announce meaningful state changes.Use a polite live region with restrained updates.
  2. Keep a mobile Downloads link.Preserve the highest-value route in the header.
  3. Repair small-text contrast.Raise two muted colors and add a website skip link.

Method and scope

Evidence, not a screenshot critique.

The audit combined rendered-flow testing with source review so visual findings could be checked against state, data, and telemetry behavior.

What was reviewed

  • Dashboard at its configured 420 × 640 production window.
  • Settings, history, export, and About/Privacy flows, including dialog bounds and focus return.
  • Product website at 1280 × 720 and 390 × 844.
  • Status logic, chart semantics, monitoring pause behavior, consent code, and native tray/window behavior.
  • Spot contrast checks for primary muted/status colors.

Limits of this audit

  • Native builds were not packaged or installed on macOS, Windows, and Linux.
  • OS notification delivery, tray menus, native save dialogs, code signing, and installer reputation were reviewed from source/configuration, not end-to-end.
  • No full VoiceOver/NVDA session or automated WCAG conformance scan was run.
  • Download URLs, release artifacts, privacy law compliance, and production analytics requests were not externally verified.
  • Findings are a heuristic product audit, not legal advice or a formal accessibility certification.
